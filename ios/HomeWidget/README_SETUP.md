# Home-screen widget — Xcode setup (when you have Mac access)

You don't own a Mac, so this guide also covers **how to get Mac access
for the one afternoon you need it**. Once the Xcode setup is done, the
widget builds and signs through CI (Codemagic / GitHub Actions) without
needing the Mac again until something fundamental in the project file
changes.

> The Swift code, Info.plist, and Dart-side sync service are already in
> the repo. This document is purely the Xcode-side wiring — and the
> options for actually getting in front of an Xcode instance.

---

## Part 1 — Getting Mac access (pick one)

For App Store submission you'll eventually need Xcode regardless of
this widget. Options for solo devs without owning a Mac:

| Option | Cost | When to pick it |
|---|---|---|
| **Used Mac mini M1** | $400-500 one-off | You're committed to iOS long-term. Cheapest TCO. |
| **MacInCloud / MacStadium (managed)** | ~$30/month | You need a few hours every couple of weeks. Pay as you go. |
| **Borrow a friend's MacBook for one afternoon** | $0 | You only need to do the one-time target setup. CI handles the rest. |
| **GitHub Actions `macos-latest` runner** | Free for ~2000 mins/mo | Build + sign automation only — *you cannot edit Xcode files there*. |
| **Apple's Self Service Repair Mac mini rental** | varies | Not actually available, ignore — listed because people google for it. |

**My recommendation**: borrow a Mac for one afternoon, do all the
Xcode-target setup steps in this doc + the Live Activity README in one
sitting, push the resulting `.xcodeproj` changes, then use Codemagic
for ongoing builds.

## Part 2 — Xcode setup for the home widget

Requires Xcode 14+. iOS deployment target 16.0+ for the widget.

### 1. Add the Widget Extension target

1. Open `ios/Runner.xcworkspace` in Xcode.
2. **File → New → Target…**
3. Pick **Widget Extension**, click Next.
4. Fill in:
   - **Product Name**: `MgbHomeWidget`
   - **Bundle Identifier**: `<your Runner bundle id>.MgbHomeWidget`
   - ❌ **Include Live Activity** (you already have a separate
     `MyGymBroLiveActivity` target for that)
   - ❌ Include Configuration Intent
   - **Embed in Application**: `Runner`
5. Click **Finish**. Activate the scheme if prompted.

Xcode will create a folder `ios/MgbHomeWidget/` with autogenerated
files. **Delete the autogenerated `.swift` and `Info.plist`** — we use
ours from `ios/HomeWidget/`.

### 2. Add our files to the new target

In Finder:

- `ios/HomeWidget/MgbHomeWidget.swift`
- `ios/HomeWidget/Info.plist`

For each:

1. Drag into Xcode under the `MgbHomeWidget` group.
2. Add to targets: ✅ **MgbHomeWidget only**.
3. For the Info.plist, set the target's
   **Build Settings → Packaging → Info.plist File** to
   `HomeWidget/Info.plist`.

### 3. Set deployment target

`MgbHomeWidget` target → **General** → Minimum Deployments → iOS 16.0.

### 4. Add the App Group capability — CRITICAL

The widget reads from a shared `UserDefaults(suiteName:)` populated by
the Dart side. Without the App Group capability set on both the main
app AND the widget extension, `UserDefaults(suiteName:)` returns nil
and the widget shows skeleton text forever.

**For the `Runner` target:**

1. Select `Runner` → **Signing & Capabilities** → **+ Capability**.
2. Add **App Groups**.
3. Add a new group: `group.com.mygymbro.widgets`
   *(must match `_groupId` in `lib/core/services/widget_sync_service.dart`).*

**For the `MgbHomeWidget` target:**

4. Repeat the same — add **App Groups**, check `group.com.mygymbro.widgets`.

> You may need to enable App Groups for your bundle id in the Apple
> Developer portal first. Xcode will offer to do this for you when you
> add the capability if your account has the right role.

### 5. Build + verify on a real device

1. Pick the `Runner` scheme (not the widget scheme).
2. Run on a real iPhone or simulator (iOS 16+).
3. Long-press the home screen → **+** → search for **MyGymBro**.
4. Add the medium widget. It should show "Start a streak" until the
   app pushes its first sync (after the first session, or after the
   home tab opens).

## Part 3 — Common breakage and how to debug

**Widget shows "Couldn't load widget"**
- The Swift file failed to compile. Open Xcode console / Issue
  navigator and read the error.
- Usually a deployment target mismatch (`StaticConfiguration` requires
  iOS 14+, `containerBackground` requires iOS 17 — the code already
  handles both via SwiftUI's API availability, but verify the target
  deployment target is set right).

**Widget shows skeleton ("Start a streak" / "Ready when you are") forever**
- App Group capability missing on one of the targets. Re-check Part 2
  step 4 — both Runner AND MgbHomeWidget need the group enabled, with
  the SAME identifier.
- Dart side is silently failing — set a breakpoint in
  `WidgetSyncService._writeAll` and check `HomeWidget.saveWidgetData`
  isn't throwing.

**Widget content is stale (won't refresh)**
- Force a refresh: kill and re-open the app, finish a session (or pull
  to refresh the home tab) → check device console for any
  `[WidgetSync] write failed:` lines.
- `WidgetCenter.shared.reloadTimelines(ofKind: "MgbHomeWidget")` is
  what the `home_widget` plugin calls under the hood. If that isn't
  firing, the `iOSName` in `WidgetSyncService` doesn't match the
  `kind` constant in `MgbHomeWidget.swift`. They must be exactly
  `"MgbHomeWidget"`.

**App Store submission**
- The widget target needs its own provisioning profile (Xcode handles
  this if you have automatic signing enabled).
- Your App Store Connect entry needs both the main app and the widget
  extension declared. Xcode's Archive flow will prompt you for this.

## Part 4 — What's already wired and works without your touch

- `WidgetSyncService` writes streak + next-focus values to the shared
  group on every `streakProvider` / `muscleRecoveryProvider` tick.
- `activeSessionNotifier` pushes a one-shot widget refresh after
  `finishSession` so today's training shows up immediately.
- The Android AppWidget reads the same keys. So the Dart side already
  has both platforms covered — the only platform-specific work is the
  Xcode target wiring described above.

If you do nothing else, Android users get a fully working widget
today. iOS users get it the moment you complete the Xcode steps.
